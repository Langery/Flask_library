### JWT 关键变化

| 维度 | X-A 前(手搓) | X-B 后(PyJWT) |
|------|--------------|----------------|
| 签名算法 | `hmac.new(b"", ..., SHA256)` 简化为占位 | `jwt.encode(payload, SECRET_KEY, algorithm='HS256')` |
| 签名 key | 字节空串(任何人都能伪造) | `app.config['SECRET_KEY']`(32 字节 dev key) |
| exp | `time.time() + 300`(5 分钟) | `time.time() + 86400`(24 小时) |
| iat | 无 | 加 `iat = time.time()`(jwt.io 显示签发时间,审计用) |
| verify | 无,后端完全不信 token | `jwt.decode(token, SECRET_KEY, algorithms=['HS256'])` 处理签名 + 过期 + 算法白名单 |
| 异常处理 | 无 | `ExpiredSignatureError` → "token 已过期" 401;`InvalidTokenError` → "token 无效" 401 |

### `require_auth` 装饰器设计

```python
@require_auth
def library_add():
    name = request.get_json().get('name')
    # g.user_id 和 g.username 已被装饰器设置,业务函数可直接用
    execute('INSERT INTO library_items (user_id, name) VALUES (?, ?)', (g.user_id, name))
    return ok({'message': 'added'})
```

被装饰的路由:
- 缺 `Authorization` header → 401 "未提供 token"
- token 过期 → 401 "token 已过期"
- token 签名错 / 篡改 → 401 "token 无效"
- 验证通过 → 业务函数执行,`g.user_id` / `g.username` 可用

### 已知限制(本轮决策,显式注明)

### uploadImg path traversal 漏洞细节

**漏洞位置(原代码 `classStore/requestway/pages_server/upload_images.py:9-23`)**
```python
filename = data.get('filename', f'img_{datetime.now().timestamp()}.jpg')
# ...
filepath = os.path.join(upload_dir, filename)
with open(filepath, 'wb') as f:  # ← 任意路径写入
    f.write(image_data.encode('utf-8'))
```

**攻击场景**:POST `/uploadImg` body `{"filename": "../../../tmp/evil.py", "image": "<malicious code>"}` → 服务端把内容写入 `/tmp/evil.py`(甚至更深)。配合 Flask DEBUG 模式 + Werkzeug debugger PIN,有可能**远程代码执行**(CVE-2024-xxxx 系列类似路径)。

**修复**:
1. `secure_filename` 第一层:把 `../`、`/`、`\` 等分隔符变成 `_`,把 `evil.py` → `evil_py`(下沉 + 去后缀)
2. `os.path.realpath` 第二层:把相对路径解析成绝对路径,杜绝 symlink 攻击
3. `filepath.startswith(upload_dir + os.sep)` 第三层:即使前两层失效,filepath 必须在 upload_dir 内,否则 400 拒绝

三层防御,任何一层单独就足够挡 known attack,叠加是 defense in depth。

### 关键设计点

#### 1. owner 隔离(横向越权防御)

```python
@library_blue.route('/library/delete/<item_id>', methods=['DELETE'])
@require_auth
def delete_item(item_id):
    row = query_one('SELECT user_id FROM library_items WHERE id = ?', (item_id,))
    if not row:
        return fail('item not found', http_status=404)
    if row['user_id'] != g.user_id:    # ← 即便 @require_auth 通过,user 也不能动别人的数据
        return fail('forbidden', http_status=403)
    execute('DELETE FROM library_items WHERE id = ? AND user_id = ?', (item_id, g.user_id))
    return ok({'deleted': item_id})
```

`@require_auth` 只验证"你是合法用户",**不验证"这条记录是你的"**;**两个机制各管一段**。DELETE 走两步:`SELECT user_id` 拿 owner,Python 比对 `g.user_id`,不等就 403。这层校验在所有"按 id 操作"的地方都不能省。

clear 不需要单独校验 —— SQL `WHERE user_id = ?` 已经天然按 user 隔离。

#### 2. sort 枚举白名单(防 ORDER BY 注入)

```python
_SORT_MAP = {
    'create_time_desc': 'create_time DESC',
    'create_time_asc':  'create_time ASC',
    'name_asc':         'name ASC',
}

sort = request.args.get('sort', 'create_time_desc')
order_by = _SORT_MAP.get(sort, 'create_time DESC')  # 非法值 fallback 到默认
sql += f' ORDER BY {order_by}'                       # 只能是白名单里的字符串
```

**ORDER BY 不能用占位符**(`?` 会被当成字符串),但也不能直接拼 `request.args.get('sort')`(用户传 `; DROP TABLE` 就挂)。**白名单是经典解法**:`.get(key, default)` 拿到的是字典里硬编码的字符串,不在字典里的 key 一律 fallback,SQL 注入字符串永远到不了 ORDER BY。

#### 3. uuid + 时间后端生成

```python
item_id = uuid.uuid4().hex
create_time = datetime.now(timezone.utc).isoformat()
```

前端不传 `id` 和 `createTime`,后端生成。**避免前端自己造 id 重复**(用 `Date.now()` 撞库),也保证时间戳不受客户端时钟影响。

#### 4. `execute_rowcount` 为什么单独一个

`db.execute()` 走 `cursor.lastrowid` 返 INSERT 的自增 id,DELETE 拿不到 rowcount(虽然 `cursor.rowcount` 是有的,但 helper 没暴露)。X-C clear 要返 `cleared: N` 告诉前端删了几条,所以补一个 `execute_rowcount()`。**两个 helper 各管一类操作**,调用方按需选。

### 2 个 endpoint 设计

| Method | Path | 功能 | 入参 | 返 |
|--------|------|------|------|-----|
| GET | `/layout/get` | 拿当前用户布局(空态返 `{items: [], updatedAt: null}`) | 无 | `{items, updatedAt}` |
| PUT | `/layout/save` | 整体覆盖(`INSERT ... ON CONFLICT(user_id) DO UPDATE`) | body: `{items: [...]}` | `{saved, updatedAt}` |

### 关键设计点

#### 1. 一行 per user(不做多行 + sort_order)

```sql
CREATE TABLE drag_layouts (
    user_id     INTEGER PRIMARY KEY,
    layout_json TEXT NOT NULL,
    updated_at  TEXT NOT NULL
);
```

**为什么不做多行?**
- 整组布局通常 10-30 个 icon,数量小
- 多行 + `sort_order` 字段 + 拖拽时 update 多行,复杂度 ↑↑
- 整组一个 JSON blob,`INSERT ... ON CONFLICT DO UPDATE` upsert 一行就够
- 代价:并发写 last-write-wins(无 diff 合并);单用户单机不构成问题

#### 2. SQLite upsert(`ON CONFLICT`)

```python
execute('''
    INSERT INTO drag_layouts (user_id, layout_json, updated_at)
    VALUES (?, ?, ?)
    ON CONFLICT(user_id) DO UPDATE SET
        layout_json = excluded.layout_json,
        updated_at  = excluded.updated_at
''', (g.user_id, layout_json, updated_at))
```

SQLite 3.24+ 支持 `ON CONFLICT ... DO UPDATE`(PostgreSQL / MySQL 也支持类似语法),等价于"先 SELECT,有就 UPDATE,没有就 INSERT",**原子操作无需事务**。`excluded.column` 引用"本应 INSERT 的值",避免二次参数化。

#### 3. size + item 上限(防滥用)

```python
MAX_LAYOUT_BYTES = 64 * 1024   # 64 KiB
MAX_ITEMS = 200

if len(items) >= MAX_ITEMS:
    return fail(f'too many items (max {MAX_ITEMS})', http_status=400)
if len(layout_json.encode('utf-8')) >= MAX_LAYOUT_BYTES:
    return fail(f'layout too large (max {MAX_LAYOUT_BYTES} bytes)', http_status=400)
```

**两层都查**:
- item 数量防"塞 1 万个空对象把表撑爆"
- 字节数防"塞 1 个超长 string 把单行撑到 MB 级"

200 item / 64 KiB 远超真实布局需求(典型 10-30 个 icon),只是兜底防滥用。

#### 4. "不主动同步"语义

后端**不**实现:
- WebSocket / SSE 推送布局变更
- 每次拖拽的 incremental PATCH
- 客户端订阅

后端**只**暴露 RESTful GET/PUT。**触发时机由前端决定**:
- 进入页面时 GET 拿初始状态
- 拖拽过程中只在本地 state 改,不调后端
- `useEffect` cleanup(组件卸载 / 路由切换)时 PUT
- 用户点"保存布局"按钮时 PUT

**这意味着**:用户连续拖拽不刷盘,意外崩溃会丢当前 session 的变更;但用户决策接受这个 trade-off(后端不被高频写打爆)。

### 验证(11 项)

```
=== 鉴权 ===
无 token GET /layout/get   → 401

=== 业务流(user A) ===
GET 初次                    → {items: [], updatedAt: null}
PUT 2 item(中文+emoji)     → saved: 2,updatedAt 写入
GET 第二次                  → 回 2 item,顺序保留,中文直存
PUT 覆盖为 1 item            → upsert 成功,GET 回 1 item

=== owner 隔离 ===
user B GET                  → 仍空

=== 边界 ===
items 非 list                → 400
items 缺字段                 → 400
items 数 700                 → 400 "too many items (max 200)"
单 item 70K name            → 400 "layout too large (max 65536 bytes)"
```

### 不在本轮范围

- **增量 sync / diff merge**:本批 PUT 整组覆盖;若要支持多设备协同,需要带 `version` / `lastModified` 的 ETag 流程
- **WebSocket 推送布局**:用户决策"不主动同步",本批不上
- **layout 模板/默认布局**:空态返空 list,前端决定渲染什么默认 icon

### 风险

- **意外崩溃丢布局**:拖拽过程不实时落库,只在 cleanup / 手动按钮落;崩溃 → 本 session 变更没刷盘
- **last-write-wins**:两设备同时编辑,后写的覆盖先写的;**用户决策接受**(单机使用为主)
- **JSON 解析兜底**:`layout_json` 损坏(手动改库)→ `json.loads` 抛 `JSONDecodeError` → GET 返空 list 而非 500,**这是有意的容错**,用户重新 PUT 即可

#### 2. 幂等 seed

```python
def seed_news_if_empty():
    existing = query_one('SELECT COUNT(*) AS c FROM news')
    if existing and existing['c'] > 0:
        return 0
    # ... INSERT 100 条
    return len(posts)
```

启服务 / 重跑 `init_db()` / 测试时反复重启都不会重复 seed。`COUNT(*) > 0` 一次性闸门,简单可靠。

#### 3. 循环 import 的解法

`db.py` 想在 `init_db()` 末尾调 `seed_news_if_empty()`,而 `seed.py` 内部用 `query_one` / `execute` —— 形成 `db ↔ seed` 循环 import。

**解法**:`init_db()` 函数内部 lazy import,模块顶层不引入:

```python
def init_db():
    # ... 建表 ...
    conn.commit()
    conn.close()

    from blueprintStore.news.seed import seed_news_if_empty  # ← 函数内 import
    seeded = seed_news_if_empty()
```

Python 模块加载时,函数体不执行,所以**函数内的 import 在被调用时才解析**,届时 `seed.py` 也早已加载完毕,无循环。

#### 4. 分页 + 模糊搜索

```python
where = ''
params = []
if keyword:
    where = 'WHERE title LIKE ?'
    params.append(f'%{keyword}%')

total_row = query_one(f'SELECT COUNT(*) AS c FROM news {where}', tuple(params))
total = total_row['c'] if total_row else 0

rows = query_all(
    f'SELECT id, user_id, title, body FROM news {where} ORDER BY id DESC LIMIT ? OFFSET ?',
    tuple(params) + (limit, offset)
)
```

- `LIMIT ? OFFSET ?` 用占位符,**limit / offset 用户可控也安全**
- `total` 单独 `SELECT COUNT(*)`,**前端可算 `Math.ceil(total / limit)` 显示总页数**
- `ORDER BY id DESC` 拿最新在前(对齐 NewsView 当前视觉顺序)

#### 5. 限流 / 边界

- `limit` 上限 50(超过 clamp),下限 1
- `page` 下限 1(0 / 负数 clamp 到 1)
- 非 int 返 400(不让 `int('abc')` 抛异常穿透成 500)
- 404 走统一 `fail('news not found', http_status=404)`
